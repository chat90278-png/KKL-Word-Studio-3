## Sprint goal

Turn the stabilized generated-document path into a clearer authoring experience and exercise it against realistic multi-page documents. The sprint must not move pagination ownership into WPF and must not duplicate Word/Preview format-selection rules.

## P0-A — Effective table-format selection UX

Current state:

- `TableElement.ReferenceTableFormatKey` persists an optional explicit profile key.
- the Properties panel already exposes `TableFormatOptions`.
- null key currently displays only `Varsayılan`, while the production resolver actually chooses the first compatible profile by table column count and then falls back to the first profile.
- with multiple compatible profiles, the user cannot see which profile is effectively active.

Acceptance:

- centralize effective table-profile selection in Application.
- `ReferenceReportContentFormatResolver` and Properties UX consume the same selector.
- persisted null key continues to mean automatic selection; no stale resolved key is written into the Domain model.
- the automatic option identifies the effective profile, for example `Otomatik — Default Table Format 1`.
- explicit profile selection continues to persist the stable profile key.
- invalid explicit keys remain rejected by `ITableFormatSelectionService`.
- imported profiles remain authoritative when present.
- Preview and Word continue to consume the same resolved table format.

## P0-B — Multi-page table pagination and continuation fidelity

Create a deterministic long-table fixture that spans at least three pages and includes:

- a real numbered table caption;
- repeated headers;
- normal detail rows;
- at least one two-serial grouped product near a page boundary;
- a group that fits on the current page;
- a group that must move intact to the next page when possible;
- enough rows to create continuation fragments.

Acceptance:

- Engine remains the pagination owner.
- Preview maps Engine pages/blocks without repagination.
- table headers repeat on continuation pages when the resolved format requests it.
- the caption appears only on the first semantic table fragment.
- the table sequence counter increments once per semantic table, not per fragment.
- grouped non-serial spans remain correct after fragment slicing.
- Serial cells remain unmerged.
- Word preserves the same semantic grouping with real `w:vMerge`.
- no group is silently duplicated or dropped at a page boundary.

## P0-C — Heading/caption keep-with-next boundary QA

Create boundary fixtures for:

- primary heading followed by paragraph;
- primary heading followed by a captioned table;
- long caption followed by a table;
- caption/table pair near the bottom of a page.

Acceptance:

- resolved `KeepWithNext` semantics are honored by Engine where supported.
- Preview and generated Word do not leave an isolated heading/caption at the bottom of a page when the following content can be moved as a unit.
- long caption measurement uses the same visible numbered caption text rendered in Preview.
- Word keeps real caption SEQ fields.

## P0-D — Canonical generated-DOCX fidelity fixture

Add one deterministic generated-document fixture/test path covering:

- A4 page geometry and margins;
- primary/secondary headings;
- body text;
- automatic numbered table caption;
- six-column unequal widths;
- fixed table layout;
- borders and cell margins;
- repeated header;
- serial/quantity vertical merge semantics;
- continuation across multiple pages where applicable.

The fixture is generated by the product pipeline. Do not replace it with a hand-authored DOCX that bypasses `ReportContentBuilder` or the Word writers.
